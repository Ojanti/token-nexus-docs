# Tokens, Variables, and the Gap

`color.brand.primary` in code. `Color/Brand/Primary` in Figma. `color-brand-primary` in CSS. These are three names for the same concept, forced into three different syntaxes by three different tools. This naming gap is not a minor inconvenience - it is the core reason automation alone consistently fails to bridge design and engineering token systems.

<ImagePlaceholder label="Screenshot: side-by-side token name comparison - GitHub JSON vs Figma Variables panel" ratio="16/9" />

## Why the names diverge

Token naming conventions in code follow the constraints of their host format. JSON uses dot-notation for nested keys. CSS custom properties use hyphens and cannot contain dots. Figma variable names use forward slashes to represent hierarchy, because slashes create the group nesting visible in the Variables panel. A <TypeBadge type="COLOR" /> token that is perfectly named in JSON will look wrong in CSS and look different again in Figma - not because anyone made a mistake, but because each tool has its own structural rules.

## Why this breaks automation

A naive synchronization script that tries to match `color.brand.primary` to `Color/Brand/Primary` has to make assumptions about casing, delimiter substitution, and hierarchy depth. Those assumptions hold until someone on your team renames a token for legitimate reasons - a refactor, an API change, a rebranding - and suddenly the automated match points at the wrong variable, or nothing at all. Silent failures are the worst kind.

## What mapping adds

The mapping layer in TokenNexus exists specifically to absorb this gap. Instead of relying on a naming convention both sides have to follow, you record the relationship once, explicitly: "this Figma variable corresponds to this token." The mapping is saved. On every subsequent sync, TokenNexus reads the mapping - not the names - to know where to look. A rename in your token file does not break the sync; it produces a diff you can review.

<Callout variant="why">
  The naming problem cannot be solved by choosing a "better" convention. Every tool in the chain has its own rules, and those rules are not going away. A persistent mapping is the only architecture that survives real-world token evolution without silent mismatches.
</Callout>
