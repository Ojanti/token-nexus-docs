# Mapping as a Contract

A mapping is a deliberate, saved decision: "this Figma variable corresponds to this design token." It is not a guess the plugin makes on each sync - it is a record you create once, stored with your file, consulted every time. When the token value changes in GitHub, the mapping tells TokenNexus exactly which Figma variable to update. No inference. No re-matching. No opportunity for drift.

<ImagePlaceholder label="Screenshot: Mapping panel showing saved variable-to-token pairs" ratio="16/9" />

## What a mapping record contains

Every mapping entry stores four things: the Figma variable ID, the Figma variable name at the time of mapping (for display), the token file path, and the token key path within that file. The variable ID is the stable anchor - names can change on either side, and TokenNexus shows you a warning rather than silently guessing.

## Why persistence matters

Without a saved mapping, every sync session would require you to re-establish which variable corresponds to which token. That is the same manual labor every time, plus the ongoing risk of making a different decision than you made last time. A persisted mapping means the work you did during setup pays forward indefinitely.

<Callout variant="tip">
  Treat your mapping file as part of your design system's source of truth. If your team uses version control for Figma or shares plugin data, exporting and committing the mapping JSON means any team member can restore the full mapping state without re-doing the work from scratch.
</Callout>
